# Thesis Proposals: Cost-Effective Training of High-Quality Language Models

Two self-contained proposals, each runnable on free or under-$50 compute with
small models. Proposal A is recommended because it builds on the solver method
already in this repository and fills a verified gap in the literature. Proposal B
is a strong, simpler alternative that answers a question with no published number.

Both connect to a verified research finding: data quality is now a measurable
scaling-law dimension, expressible as a multiplier on the token budget
(arXiv:2510.03313, arXiv:2410.03083), yet no published work ranks the cost levers
by quality-per-dollar, and no published work gives a controlled QLoRA-vs-full
fine-tuning quality number on a single 16GB GPU.

---

## Proposal A (recommended): Is zero-noise solver-generated data more efficient per dollar than real or distilled data?

### Question
For a task with a known closed-form solver, does training data generated by that
solver (zero label noise, controllable difficulty) reach a target accuracy with
fewer tokens and fewer dollars than (a) real labeled data and (b) data distilled
from a larger language model?

### Why it matters
Frontier labs already lean on synthetic data (over 98% of Nemotron-4-340B
alignment data was synthetic; arXiv:2406.11704), and quality-aware scaling laws
say data quality trades against model size. But no one has measured the
efficiency of perfectly-correct solver data against alternatives on a common
quality-per-dollar axis. This is directly fundable and directly extends the
solver pipeline in this repo.

### Hypothesis
Solver-generated data has a higher effective-token multiplier than real or
distilled data, so it reaches the same accuracy at a lower token budget and lower
dollar cost, with the gap largest in the low-data regime.

### Method
- Model: one small open base (for example Qwen2.5-1.5B or Llama-3.2-1B).
- Task: an inductive-reasoning family with a known solver (reuse the solvers in
  src/solvers, for example the gravitational or unit-conversion family).
- Three data sources at matched token budgets: solver-synthetic (zero noise),
  real labeled rows, and language-model-distilled rows (noisy).
- Train with 4-bit QLoRA, identical hyperparameters, several token budgets.
- Fit accuracy vs token budget for each source; read off the effective-token
  multiplier and accuracy-per-dollar.

### Baselines
Real data and distilled data, same model, same budgets.

### Metrics
Held-out task accuracy, effective-token multiplier per source, accuracy per
dollar (token budget times measured GPU cost).

### Compute
Free Kaggle or Colab T4 (QLoRA on a model at or below 1.5B fits 16GB). Total
under $50 even if a paid spot GPU is used for the largest runs.

### Expected contribution
A measured quality-per-dollar ranking of data sources, plus the first controlled
evidence on solver-generated zero-noise data efficiency. Fills research gap 2
(no dollar-per-quality ranking) and gap 3 (scaling laws beyond the fitted regime).

### Risks and mitigations
- Task generality: run on at least two solver families to show it is not family
  specific.
- Distillation fairness: filter distilled data by answer correctness so the
  comparison is quality, not just noise.
- Contamination: hold out a freshly generated test split.

## Proposal B (alternative): The missing number, how much accuracy does 4-bit QLoRA actually cost vs full fine-tuning on a single 16GB GPU?

### Question
Across small models and several tasks, what is the accuracy gap between full
fine-tuning, LoRA in bf16, and 4-bit QLoRA, measured on a single 16GB GPU, and
where does that gap become non-negligible?

### Why it matters
QLoRA (arXiv:2305.14314) claims near-parity with full fine-tuning, but the deep
research pass found no published controlled number for the quality loss on a
single 16GB GPU across a model-size and task sweep. Students and small labs make
this choice constantly with no data to guide it.

### Hypothesis
4-bit QLoRA loses a small, characterizable number of accuracy points vs full
fine-tuning at small scale, and the gap shrinks as model size grows and as the
task is more in-distribution for the base model.

### Method
- Models: a size sweep (for example 0.5B, 1B, 3B) that fits 16GB.
- Three training modes: full fine-tuning, LoRA (bf16), QLoRA (4-bit), with
  matched data, steps, and tuned learning rates per mode for fairness.
- Tasks: three to five standard small benchmarks.
- Record accuracy, peak memory, throughput, and cost per run.

### Baselines
Full fine-tuning is the upper-bound reference; the base model with no tuning is
the lower bound.

### Metrics
Accuracy delta per mode, peak GPU memory, tokens per second, dollars per run.

### Compute
Free T4 or P100. Full fine-tuning of sub-1B models fits 16GB; larger sizes use
QLoRA only, which is the point. Under $50.

### Expected contribution
The first controlled accuracy-vs-memory-vs-cost chart for full vs LoRA vs QLoRA
at student scale, a practical guide for when 4-bit is effectively free. Fills
research gap 1.

### Risks and mitigations
- Hyperparameter fairness: tune the learning rate separately per mode and report
  the search.
- Task choice: use established benchmarks to avoid cherry-picking.

---

## Recommendation

Start with Proposal A. It is more novel, it builds on the solver code you already
understand, and it produces a quality-per-dollar result that no one has published.
Proposal B is the safer fallback and can be run as a smaller companion study,
since both share the same training and evaluation harness.
